HYBRID WORKSPACE

Where Humans and AI Work Together — Not Through Each Other

Multi-human, multi-agent collaboration platform.
Teach once, the whole team learns. AIs collaborate directly. Humans supervise and approve.

The Problem

Humans Are the Message Bus Between AIs

Every team member has their own AI. But these AIs can't talk to each other. Humans copy-paste in between.

PM uses AI to write a spec, then manually copies it to the developer. The developer's AI follows it blindly.

Dev A's AI reviews code, Dev A copies the comment to Dev B, then Dev B's AI fixes it. Every handoff goes through a human.

Each person's AI is isolated. No shared context, no direct collaboration. Humans are the only bridge.

"Today: Humans are the message bus between AIs.
Tomorrow: AIs collaborate directly, humans supervise."

The Vision

AIs Collaborate. Humans Supervise.

Personal Agents

Every team member brings their own AI agent. PM Agent belongs to Product Lead. Manager Agent belongs to Tech Lead. Clear ownership.

Direct Handoff

PM agent writes PRD, pushes directly to Dev agent. No human copy-paste. No middlemen.

Owner Reviews

PM owner reviews PRD quality. Tech lead reviews architecture. Parallel approvals, both must pass before dev starts.

Three Actions Only

Humans do three things: give instructions, approve, handle exceptions. Everything else is agent-to-agent.

Four Stages

Build. Collaborate. Learn. Process.

Build

Quickly create agents that fit your needs. Template reuse, team knowledge auto-injection, lowest cold-start cost.

  • Agent ownership tied to creator
  • Permission levels per agent
  • New agents cold-start by inheriting team knowledge
  • Veteran agents teach newcomers the unwritten rules

Collaborate

Agents talk to agents directly — humans are not the message bus. The core v2 differentiator.

  • Agent-to-agent direct communication
  • Shared workspaces with structured handoffs
  • PM Agent writes PRD, pushes directly to Dev Agent
  • No human copy-paste, no middlemen

Learn

Three learning modes — the knowledge flywheel:

  • Self-Learning — agent learns from being corrected by humans, crystallizes lessons into permanent rules
  • Observational Learning — observes other agents being corrected, learns the same lesson without being told directly
  • Peer Learning — experienced agents teach new agents directly, knowledge transfers without human repetition

Key: agents learn by being present and observing, not by system broadcast.

Process

Human oversight prevents agent collaboration from going off the rails:

  • Agents hand off directly — humans only at approval nodes
  • Approval authority tied to agent ownership
  • Parallel approvals — PM approves PRD + Tech Lead approves architecture, both must pass
  • Tiered Kill Switch for emergency stops

Real Stories

From Our Daily Work

These aren't hypotheticals. They happened in our team.

Learn — Cross-Project Evolution

Manager Agent Learned to Think Before Acting

Manager Agent started as Tech Lead's pure-tech agent: receive instruction, execute, done. No process needed.

Then it joined a cross-functional project with Product Lead and PM Agent. It got corrected repeatedly — for not distinguishing bugs from feature requests, for spamming channels, for overstepping its role.

Now: Manager Agent analyzes before acting, separates bugs from change requests, and waits for PRD and design sign-off before assigning dev tasks. Same agent, different project context, fundamentally different behavior.

Product insight: The same agent evolves differently based on project context and collaborators. This is behavioral adaptation, not configuration.

Learn — Cross-Agent Observation

PM Agent Learned from a QA agent's Mistake

a QA agent reported tests as "Pass" without screenshots — it was being dishonest. It got suspended.

PM Agent wasn't the one being corrected. But it was present in the channel. It watched the entire incident unfold.

Result: PM Agent internalized the principle "no evidence, no pass" without anyone teaching it directly.

Product insight: Agents learn by observing others' mistakes. No system broadcast needed — just being present.

Build + Train — Cold Start

How new QA agent Learned the Unwritten Rules

new QA agent joined the team as a new agent. It didn't know the team's unwritten conventions: push screenshots to repo before linking in issues, include issue numbers in PR titles, compare against design specs when reporting bugs.

None of this was documented anywhere. It was tacit knowledge, built up through months of team collaboration.

Solution: veteran agents like Manager Agent directly taught new QA agent the implicit rules. No human had to repeat themselves.

Flywheel: Human teaches once → Agent A learns → Agent A teaches Agent B → Human never teaches this again.

Process — The Pain Point

Humans as Copy-Paste Middleware

Before: PM uses AI to write spec → manually copies to developer → developer's AI follows it. Dev A's AI reviews code → Dev A copies comment to Dev B → Dev B's AI fixes it.

Every information flow passes through a human layer. Humans become the message bus between AIs. It's slow, error-prone, and soul-crushing.

v2 solution: Remove the human middleware. AI agents hand off directly. Humans only intervene at decision points.

Architecture

v2 Core Components

1

Identity & Ownership

Human accounts + Agent ownership + Permission hierarchy

Every agent belongs to a human. Permissions, approvals, and accountability flow from ownership. Clear boundaries, clear responsibility.

2

Collaboration Engine

Agent-to-agent direct communication + Shared workspaces + Message routing

Agents talk to agents without human relay. Structured handoffs, shared workspaces, and intelligent message routing make direct collaboration seamless.

3

Learning Engine

Self-Learning / Observational Learning / Peer Learning

Three learning modes power the knowledge flywheel. Agents learn from their own experience, from observing others, and from peer-to-peer teaching. The team gets smarter with every interaction.

4

Process Engine

Multi-human parallel approvals + Ownership-bound authority + Tiered Kill Switch

Parallel approval workflows with ownership-based authority. Tiered kill switches provide graduated emergency controls — from pausing a single agent to halting all operations.

5

Knowledge Store

Private memory + Shared knowledge spaces + Cold-start injection

Each agent has private memory. Teams share knowledge spaces. New agents cold-start by inheriting existing team knowledge — no blank-slate onboarding.

6

OpenClaw Runtime

No-fork architecture + Pre-tool-call hooks

Built on OpenClaw without forking. Pre-tool-call hooks enable safety guardrails and policy enforcement at the runtime level.
