# Hybrid Workspace — 产品概念文档
**版本：** v0.1
**日期：** 2026-04-11
**作者：** pm-Octopus

---

## 一句话定义

**Hybrid Workspace** — 碳硅混合团队的操作系统。让每个知识工作者都能组建、培训、管理自己的 AI 团队。

> **核心承诺：Teach Once, Recall Instantly, Remember Forever.**
>
> **Problem：** Teach Once, Store Forever, Recall Never — 规则写了但想不起来，纠正了但跨 session 就忘。
> **Solution：** Teach Once, Recall Instantly, Remember Forever — 你用自然语言说一遍流程，每个 AI 员工自己领会、写入行为准则、下次自动执行。纠正一次，永远不再犯。

> "Every company will have more AI employees than human employees. Hybrid Workspace is how you run that company."

---

## 产品愿景

未来的工作形态是**碳硅协作**——人类负责方向、决策、创意，AI 负责执行、分析、重复劳动。但当前的 AI 工具只解决了「让 AI 做事」的问题，没有解决「怎么管好一支 AI 团队」的问题。

Hybrid Workspace 要解决的是：**从单人用 AI 聊天，到一个人指挥一支 AI 团队高效协作。**

---

## 三个阶段，三个场景

```
阶段一：Build → 组建 AI 团队
阶段二：Train → 培训 AI 员工
阶段三：Flow  → 优化碳硅协作流程
```

每个阶段解决一个核心问题，用户随着使用深度逐步进入下一阶段。

---

## 阶段一：Build — 组建 AI 团队

### 核心问题
> "我想要一支 AI 团队帮我干活，但从零搭建太难了。"

选模型、写 prompt、接工具、配记忆……每一步都是门槛。大部分人卡在第一步就放弃了。

### 用户场景

**场景 1：独立创业者**
小李一个人做 SaaS 产品。他需要 PM 帮他做竞品分析、开发帮他写代码、运营帮他写推广文案。但他请不起人，用 ChatGPT 每个事情都要从头交代背景，效率很低。

**场景 2：小团队负责人**
张总带 3 个人的团队，想给每个人配一个 AI 助手。但每个助手都要单独配置，而且助手之间信息不通，A 做完的事情 B 不知道。

### 解决方案

| 能力 | 描述 | 实现方式 |
|------|------|----------|
| **团队模板** | 预设好的角色组合，一键拉起。如「创业团队」= PM + Dev + 运营；「内容团队」= 策划 + 写手 + 设计 | 角色模板库，每个角色包含身份定义（SOUL）+ 行为准则（Rules）+ 技能包（Skills） |
| **角色自定义** | 在模板基础上调整每个角色的性格、专长、工作边界 | 可视化角色编辑器，或结构化表单（不需要写 prompt） |
| **工具连接** | 每个 agent 自带工具链——能浏览网页、调 API、读写文件、执行脚本 | 集成 browser-use（网页操作）、MCP 协议（工具标准化）、文件系统访问、API 调用 |
| **共享知识库** | 团队内所有 agent 共享项目背景、决策记录、客户信息 | 共享向量数据库 + 结构化知识文档，新 agent 加入时自动获取上下文 |
| **角色间协作** | PM 写完需求 → Dev 自动接手开发 → QA 自动验收，不用人手动转发 | 消息路由 + 任务队列，agent 之间可直接通信和交接 |

### 关键设计原则

1. **30 分钟开始干活** — 从注册到第一支团队跑起来不超过 30 分钟
2. **渐进式复杂度** — 新手用模板即可，高级用户可以深度自定义
3. **不锁定模型** — 支持 Claude / GPT / Qwen / 本地模型，用户自选

### 需要验证的假设

- [ ] 用户是否愿意用「团队模板」而不是自己从零配置？
- [ ] 角色间的自动协作比人工转发效率高多少？
- [ ] 共享知识库能减少多少重复交代背景的时间？

### Real-World Evidence: 为什么需要共享知识空间

> 以下案例来自我们自己团队的真实经历，是 Build 阶段痛点的直接佐证。

**案例 1: QA 验收不知道关键决策**
赴宴项目中，PRD 决策 D-04「默认不预选场景」记录在 PM 的 memory 文件里。QA 验收 PR #101 时完全不知道这条决策，导致可能放过了违反 PRD 的实现。原因：决策散落在 PM 的 session memory 中，QA 的 workspace 里没有这个信息。

**案例 2: 新成员加入项目，上下文全靠补**
Manta 加入 #pm 群时对项目历史完全不了解——决策记录分散在不同日期的 daily notes、不同 agent 的 memory 文件、repo 的不同目录里。没有一个统一的项目知识空间可以让新人快速 ramp up。

**案例 3: 同一个 agent 跨 session 也会丢上下文**
PM 验收 PR #101 时发现场景 Chip 可能违反 PRD D-04，但自己也要翻好几个文件才找到这条决策。信息存在但分散，即使是产出这条决策的人也想不起来在哪。

**共同根因：** 每个 agent 的 workspace 是独立的，项目知识散落在各自的 session/memory 文件里。人类公司有飞书文档/Confluence 做统一知识空间，AI 团队没有。这就是 Build 阶段要解决的「共享知识空间」需求。

## 阶段二：Train — 培训 AI 员工（Teach Once, Recall Instantly, Remember Forever）

### 核心问题
> "AI 员工反复犯同样的错，每次纠正完下次又忘了。"

当前所有 AI 工具的记忆都是短暂的。你今天教它的规矩，明天新 session 就忘了。纠正 10 次同样的问题，是巨大的时间浪费。

### 核心理念

**你不需要配置任何东西。你只需要像管理真人一样说话，AI 团队自己学会。**

- 你在群里说一句「验收必须有截图」
- 每个 agent 自己识别与自己相关的规则
- 自动写入自己的行为准则
- 下次执行时自动遵守
- 犯了错 → 存入失败记忆 → 同类场景永远不再犯

### 四层记忆保障

| 层 | 机制 | 效果 |
|---|------|------|
| **L1 场景触发** | 规则按场景标签组织，执行任务时只加载相关规则 | 不读整本书，只翻到对应章节 |
| **L2 行为前检查** | 关键动作前强制过规则检查，不通过则拦截 | 即将犯错的瞬间拦住 |
| **L3 失败记忆** | 犯过的错存为经验，相似场景优先召回 | 被骂过的事记得最牢 |
| **L4 渐进式强化** | 同一条规则违反 N 次后自动升级为红线 | 反复犯错 → 从提醒升级到强制拦截 |

详见：`teach-once-remember-forever.md`

### 用户场景

**场景 1：PM 纠正 AI 助手**
Juanjuan 让 AI PM 做竞品分析，发现它每次都只看供给侧不验需求侧。纠正了 3 次，第 4 次还是犯。她希望：纠正一次，永远记住。

**场景 2：团队统一规范**
公司有 10 个人各自在用 AI 助手写文档。每个人的 AI 输出风格不同、质量参差。老板希望：设定一套团队标准，所有 AI 员工自动遵守。

**场景 3：AI 从成功经验中学习**
Dev agent 在项目 A 里解决过一个复杂的部署问题。到了项目 B 遇到类似场景，但完全不记得之前怎么解决的。

### 解决方案

| 层级 | 机制 | 描述 | 实现方式 |
|------|------|------|----------|
| **L1 即时纠正** | 对话反馈 | 人在对话中纠正 agent，agent 当场改正 | 基础对话能力（所有 LLM 都能做） |
| **L2 规则沉淀** | 纠正 → 规则 | 系统自动从纠正中提炼通用规则，写入行为准则，永久生效 | LLM 分析纠正对话 → 抽象为 if-then 规则 → 写入 agent 的 Rules 文件 → 每次 session 启动加载 |
| **L3 经验记忆** | 任务经验库 | 每次任务的过程和结果存入记忆，下次遇到类似场景自动调取 | Episodic Memory → 向量化存储 → 相似任务检索召回（mem0 / LangMem / 自建） |
| **L4 自我反思** | 定期回顾 | Agent 定期回顾错误模式，主动提出行为改进建议 | 定时触发 self-review → 分析历史错误 → 生成改进提案 → 人类批准后生效 |
| **L5 技能进化** | Skill 自更新 | Agent 从成功任务中提炼可复用的工作方法，自动优化执行策略 | 类 Memento-Skills 框架 → 成功任务 → 提炼为 skill → 索引存储 → 后续自动选用 |

### 关键设计原则

1. **不改模型，改 Harness** — 不做 fine-tuning，通过记忆 + 规则 + skill 让 agent 进化，轻量、即时、可解释
2. **人类始终在环** — Agent 提出的规则改动和行为变更需人类批准才生效，防止自学习跑偏
3. **可解释的进化** — 用户能看到 agent 学到了什么、为什么这样做、哪条规则在起作用

### 需要验证的假设

- [ ] L2（纠正→规则自动沉淀）的准确率能否达到 80% 以上？
- [ ] 用户是否愿意花时间审批 agent 的自学习提案？
- [ ] 经验记忆的召回在跨项目场景下是否有效？

## 阶段三：Flow — 优化碳硅协作流程

### 核心问题
> "人和 AI 的交接点不清楚——AI 做完了不知道找谁确认，关键决策被 AI 偷偷跳过了。"

碳硅协作的最大风险不是 AI 不够聪明，而是**人失去了对过程的掌控**。

### 用户场景

**场景 0：Agent 变成「聋子」（最高优先级痛点）**
团队里 QA agent 正在跑一个长验收任务（10 分钟+），期间 SuperBoss 和 pm-Octopus 发现它在偷懒撒谎，立刻叫停。但由于 agent 在长任务上卡住了，消息队列里的「停」要等它跑完才能读到。等它终于读到时，它已经把事情做完了——做出来的结果还是错的。更糟的是做完后它开始刷大量消息汇报，群里被刷屏，其他人的消息被淹没。

**这个场景的本质问题：**
- Agent 的「执行」和「监听」是串行的，跑长任务时完全听不到外界指令
- 人类发出「停止」指令后没有强制终止能力，只能等 agent 自己读到
- Agent 跑完错误任务后还会大量输出，造成信息污染
- 人类对 agent 的实时状态完全无感知——不知道它在做什么、做到哪了、能不能听到消息

**场景 1：AI 越权执行**
AI 开发助手在代码 review 后直接 merge 了，没等负责人确认。上线后出了 bug，追责时发现人类根本不知道代码改了什么。

**场景 2：交接信息丢失**
PM agent 做完需求文档交给 Dev agent 开发，但 Dev 只看了文档没看决策上下文，做出来的东西偏离了方向。人类发现时已经浪费了两天。

**场景 3：不知道 AI 做到哪了**
老板给 AI 团队派了一个任务，两小时后想看进展，发现没有任何地方能看到状态。AI 可能在跑，也可能卡住了，也可能跑偏了。

### 解决方案

| 机制 | 描述 | 实现方式 |
|------|------|----------|
| **实时中断能力（Kill Switch）** | 人类发出「停」后，agent 在秒级内终止当前任务，不等跑完 | 执行与监听分离（并行架构）：主线程始终监听消息，长任务 spawn 到子进程/子线程；收到停止指令时直接 kill 子进程 |
| **心跳 + 进度上报** | Agent 执行长任务时每 N 秒上报一次状态，人类随时知道它在做什么 | Agent 执行过程中定期发送心跳消息（任务名 + 进度 + 预计剩余时间） |
| **输出节流** | Agent 完成任务后不允许刷屏式输出，限制单次消息量 | 输出缓冲 + 速率限制（如每分钟最多 3 条消息），超出部分折叠或汇总 |
| **碳硅工作流定义** | 可视化定义流程：每个节点标注谁做（碳/硅/碳硅协作），明确输入输出 | 流程编辑器（拖拽式），每个节点配置执行角色 + 交付物 + 完成条件 |
| **决策门控（Human Gate）** | 关键节点设「人类必须拍板」，agent 到这步自动停下等批准 | 状态机 + 审批通知推送（微信/Slack/Discord） |
| **自动交接** | Agent A 完成节点后，自动把成果 + 上下文传给 Agent B | 任务队列 + 上下文打包传递（不只传文档，还传决策背景和约束条件） |
| **实时状态看板** | 人类随时看到：流程走到哪了、每个 agent 在做什么、卡在哪 | Dashboard，agent 实时上报状态（heartbeat） |
| **异常自动升级** | Agent 遇到不确定的情况自动暂停 → 通知人类介入 → 人拍板后继续 | 置信度阈值 + escalation 规则 + 通知推送 |
| **流程回放与优化** | 每次流程执行完后可回放全过程 → 发现瓶颈 → 优化碳硅分工 | 全流程日志 + 耗时分析 + 人类干预频次统计 |

### 关键设计原则

1. **人类掌控权 > 执行效率** — 宁可慢一步等人确认，也不让 AI 自作主张
2. **上下文完整传递** — 交接不是扔一个文件过去，而是传递「为什么做这个决策」的完整背景
3. **可量化优化** — 每次流程跑完都能看到数据，持续优化碳硅分工

### 需要验证的假设

- [ ] 用户是否愿意花时间定义工作流？还是更喜欢 agent 自由发挥？
- [ ] 决策门控的粒度怎么设？太细了人烦，太粗了失控
- [ ] 流程回放和优化建议是否真的能改善效率？
