# Teach Once, Remember Forever — Agent 长期行为记忆设计

**日期：** 2026-04-11
**作者：** pm-Octopus
**状态：** 概念探索

---

## 核心问题

> 人类纠正了 agent 一次，怎么保证它在所有未来的 session、所有类似场景中都不再犯？

这是 Hybrid Workspace 的核心挑战，也是最大的卖点。当前所有 AI agent 系统都没有真正解决这个问题。

---

## 为什么现有方案不够

### 当前做法：写进 AGENTS.md
把每次纠正提炼为规则，写入一个大文件，每次 session 启动时加载。

**三个致命问题：**

1. **规则膨胀** — 文件越来越长（我们的 AGENTS.md 已有 20+ 条踩坑记录），关键规则被淹没在大量文本中
2. **读了不等于记住** — Agent 读入了全部规则，但在关键时刻不一定想得起来。就像新员工读了100页员工手册，上班第一天还是犯错
3. **上下文窗口竞争** — 规则越多，占用 context window 越多，留给实际任务的空间越少

### 类比人类组织
公司也有这个问题。规章制度写了一大本，员工不读、读了不记、记了不做。真正有效的是什么？

- **师傅带徒弟** — 在具体场景中教，不是读手册
- **犯错的记忆最深** — 被骂过的事不会再犯
- **关键时刻的提醒** — 不是一直念叨规则，而是在即将犯错的那个瞬间拦住

---

## 解决思路：四层记忆保障机制

不是选一个方案，而是四层叠加，层层递进。

### Layer 1：场景触发式规则（精准加载）

**原理：** 规则不再平铺成一个大文件，而是按场景标签组织。Agent 执行验收任务时只加载验收规则，做产品分析时只加载分析规则。

**效果：** 相当于「打开对应的操作手册那一章」，而不是读整本书。

**实现思路：**
- 每条规则打标签：适用角色、适用场景、触发条件
- Agent 接到任务时，根据任务类型自动召回匹配的规则子集
- 规则分级：红线规则（必须加载）vs 建议规则（按需加载）

**解决的问题：** 规则膨胀、context window 浪费

**风险：** 标签打错 → 该触发的规则没触发。需要回归测试机制。

---

### Layer 2：行为前检查点（执行前拦截）

**原理：** Agent 每次执行关键动作前，强制过一道「规则检查」。不是提醒，是拦截——检查不通过就不能继续。

**效果：** 相当于「代码的 lint + CI」——不是事后 review，是提交前自动检查。

**关键检查点：**
- 验收报告提交前 → 检查每一项是否有截图
- Bug 修复 PR 提交前 → 检查是否关联了 Issue
- 产品分析结论输出前 → 检查是否覆盖了供给侧+需求侧+付费意愿
- 停止指令后的任何输出 → 检查是否收到过停止指令

**实现思路：**
- 定义「检查点」：哪些动作需要检查、检查什么条件
- 检查失败 → 阻断输出 + 告知 agent 原因 + 通知人类
- 检查点可配置，用户可以增减

**解决的问题：** Agent 知道规则但执行时跳过（最常见的失败模式）

**风险：** 检查点太多会拖慢速度。需要只在关键动作设检查点。

---

### Layer 3：失败记忆优先召回（经验学习）

**原理：** 每次犯错的经历存成「失败记忆」，下次遇到相似场景时，优先召回「上次在这里栽过」。

**效果：** 最接近人类的学习方式——被骂过的事记得最牢。

**记忆结构：**
```
{
  "scene": "验收 PR #37",
  "mistake": "只验最近 PR，没覆盖累积改动",
  "consequence": "验收后发现 6 个新 bug",
  "correction": "验收范围 = PR 改动页 + 全流程回归",
  "timestamp": "2026-04-03",
  "severity": "high"
}
```

**召回机制：**
- 当 agent 接到新的验收任务时
- 搜索历史失败记忆中与「验收」相关的条目
- 将最相关的 2-3 条失败经验注入 context：「提醒：上次做验收时只验了最近的 PR，漏了累积改动，结果发现 6 个 bug。这次要覆盖全流程。」

**解决的问题：** 跨 session 的经验传承

**风险：** 相似度匹配不准 → 误召回不相关的经验，或者遗漏真正相关的经验。

---

### Layer 4：渐进式强化（自适应升级）

**原理：** 统计每条规则被违反的次数。同一条规则被纠正 N 次后，自动升级为更高优先级。

**升级路径：**
```
建议规则 →（违反 2 次）→ 重要规则 →（违反 3 次）→ 红线规则（执行前强制检查）
```

**效果：** 系统自动识别「这个 agent 老在这个问题上犯错」，从温和提醒逐步升级到强制拦截。就像公司对反复犯错的员工，从口头提醒到书面警告到写进绩效。

**实现思路：**
- 每条规则维护违反计数器
- 达到阈值时自动升级规则级别
- 升级时通知人类确认（防止误升级）
- 如果长期（如 30 天）未再违反，可以降级回原始级别

**解决的问题：** 不需要人类手动判断哪些规则该升级为红线

**风险：** 可能因为环境变化导致误升级。需要人类有最终否决权。

## 四层叠加后的效果

以「验收必须有截图」这条规则为例：

```
Juanjuan 说：「验收必须有截图」
         ↓
Layer 1：规则打标签 [角色:QA, 场景:验收]
    → 下次 QA 做验收时自动加载这条规则
         ↓
Layer 2：设检查点 [验收报告提交前 → 检查截图字段]
    → QA 提交没有截图的报告时被拦截
         ↓
Layer 3：如果 QA 仍然被拦截（试图绕过）
    → 这次失败经历存入失败记忆
    → 下次开始验收时提醒「上次你提交了没有截图的报告被拦截了」
         ↓
Layer 4：如果被拦截 3 次
    → 自动升级为红线规则
    → 每次验收开始前强制弹窗确认「你知道必须每项都有截图吗」
```

**最终效果：从人类说一句话，到 agent 永远不犯这个错。**

## 与产品概念的关系

这个机制是 **Train 阶段（培训 AI 员工）** 的核心引擎，也是 Hybrid Workspace 最大的差异化：

- 现有工具：AI 每次都从零开始，不记得上次犯的错
- Hybrid Workspace：**Teach once, remember forever**

Slide 7（Phase 2: Train）的核心信息应该围绕这个讲：

> "Your AI employees learn from every correction. One mistake becomes a permanent lesson — not just for one agent, but for the entire team."

## 开放问题

1. **四层机制的优先级？** MVP 先做哪层？建议 Layer 1（场景触发）+ Layer 2（检查点），因为效果最直接
2. **跨 agent 学习？** QA 犯的错，PM 和 Dev 是否也应该知道？共享失败记忆 vs 角色隔离
3. **遗忘机制？** 如果业务流程变了，旧规则怎么退休？需要规则过期/归档机制
4. **度量指标？** 怎么衡量「记住了」？建议：同一类错误的重犯率（目标：纠正后重犯率 < 10%）
