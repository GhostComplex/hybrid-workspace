# Hybrid Workforce — 产品规划 v0.1

> **愿景：** Build your workforce, not your tools.
> **定位：** 让任何 information worker 都能拥有并管理一支 AI 团队
> **核心信念：** 我们自己就是第一个用户，产品从真实痛点出发

---

## 产品北极星

**一个人 + 一套 Agent 团队 = 过去一个小组的产出**

衡量指标：
- 用户从"有想法"到"Agent 团队开始工作"的时间 < 30 分钟
- Agent 团队在没有人类实时干预的情况下，自主完成任务的比例 > 80%
- 用户感知："Agent 越来越好用"，而不是"每次都要从头调教"
- **用户能看到 Agent 学到了什么、什么时候学的、有没有真正生效** ← 核心信任指标

---

## 三大产品模块

```
┌─────────────────────────────────────────────────────────┐
│  模块一：Workforce Builder — 搭建你的 Agent 团队         │

├─────────────────────────────────────────────────────────┤
│  模块三：Workforce OS — 碳硅协作的运作系统               │
└─────────────────────────────────────────────────────────┘
```

## 模块一：Workforce Builder

### 要解决的核心痛点
- 搭 Agent 团队门槛极高，普通人搞不定
- 角色设计、工具选择、协作规则全靠摸索
- 每个人从零开始，经验无法复用

### 核心功能

**1.1 角色模板库**
预设开箱即用的 Agent 角色，用户按需组合：

| 角色 | 核心能力 | 典型工具 |
|---|---|---|
| 研究员 | 信息收集、市场调研、竞品分析 | browser-use、爬虫 CLI |
| PM | 需求分析、PRD、优先级决策 | 文档生成、框架分析 |
| 执行者 | 编码、内容生成、数据处理 | Code interpreter、API 调用 |
| 质检员 | 验收、审查、异常发现 | LLM-as-judge、assertion |
| EM | 进度追踪、blocker 清除、协调 | Cron、消息推送 |

**1.2 团队组合向导**
根据用户的目标，推荐角色组合：
- 输入：我想做什么（产品开发 / 市场调研 / 内容生产）
- 输出：推荐的 Agent 团队配置 + 协作规则

**1.3 工具接入市场**
Agent 可按需安装工具，安装后变成持久能力：
- 标准化工具接入（基于 MCP 协议）
- 社区贡献的 skill 包（browser-use / 小红书爬虫 / PPT 生成等）
- 工具使用记录，Agent 学会后不需要重新配置

**1.4 协作规则配置**
可视化配置团队的运作规则：
- 任务分发路径（谁给谁派活）
- 决策升级规则（遇到什么情况找人）
- 输出格式标准（文件夹结构、文档模板）

### 成功标准
> 没有 AI 工程背景的 PM，30 分钟内搭起可以开始工作的 Agent 团队

## 模块二：Workforce Trainer

### 要解决的核心痛点
- Agent 犯错不知道怎么纠正，改了 prompt 又破坏其他地方
- 好的经验无法沉淀，下次同类问题又从头来
- Agent 越用越让人不放心，而不是越用越好
- 角色边界不清，越界、越级问题反复出现

### 核心功能

**2.1 结构化 Onboarding**
每个 Agent 上岗前完成标准化入职流程：
```
角色定义 → 行为准则 → 边界清单 → 工作方式 → 记忆初始化
```
- 提供各角色的 onboarding 模板
- 入职检验：用 5 个标准任务测试 Agent 是否就绪

**2.2 规则持久化与可信机制（核心差异化）**
Agent 说"我已学会"——用户凭什么相信？

```
人给 feedback（口语化）
    → 系统解析为结构化规则
    → Agent 写入行为准则文件（强制落盘，不存在 context 里）
    → 系统自动验证写入成功
    → Agent Profile 可见：规则来源 + 学习时间 + 生效记录
    → 下次同类场景触发时，标注"依据规则 #12 执行"
```

**这是 Hybrid Workforce 与所有"AI 助手"产品的根本区别：**
别人的 AI 说完就忘，我们的 Agent 是真正在成长的——而且成长是可见的、可追溯的。

- **可信**：规则真正写入文件，系统验证，不靠 Agent 自觉
- **可见**：Agent Profile 展示所有已学规则，用户一目了然
- **可追溯**：每条规则的来源、学习时间、上次生效时间全部有记录
- **防冲突**：多 Agent 同时更新规则时，有仲裁机制避免矛盾

**2.3 反馈闭环系统**
把人的 feedback 转化为 Agent 的持久行为改变：

```
人给 feedback（口语化）
    → 系统解析为结构化规则
    → Agent 更新行为准则
    → 标注"来源于 feedback"，可追溯
    → 下次同类场景自动应用
```

痛点直指：现在的反馈是一次性的，说完就忘；我们要让每次纠正都沉淀下来。

**2.3 能力成长追踪**
可视化 Agent 的成长轨迹：
- 任务完成质量趋势
- 已掌握的方法论 / 工具 / 领域知识
- 历史踩坑记录 + 修复情况
- "这个 Agent 现在能干什么"一目了然

**2.4 记忆体系**

| 记忆层 | 内容 | 生命周期 |
|---|---|---|
| 任务记忆 | 当前任务 context | 任务结束归档 |
| 日常记忆 | 每天工作记录 | 永久 |
| 项目记忆 | 决策、进展、资料 | 永久，Git 管理 |
| 角色记忆 | 方法论、经验、准则 | 永久，持续更新 |
| 组织记忆 | 团队结构、联系人、规则 | 永久 |

**2.5 自主学习机制**
Agent 可以主动扩展自己的能力边界：
- 遇到不擅长的领域，主动搜索相关方法论并沉淀
- 定期回顾历史任务，提炼规律
- 把学到的东西写进自己的知识库，不靠人提醒

### 成功标准
> 同类错误再发生率下降 80%；新 Agent 加入后 30 分钟可独立工作

## 模块三：Workforce OS

### 要解决的核心痛点
- 不知道 Agent 在干什么，不放心但不想一直盯
- 叫停指令有延迟，Agent 收到时任务已完成（消息队列问题）
- Agent 出问题时自己发现不了，等人发现已经晚了
- Agent 之间沟通不透明，人无法追溯决策过程
- 人介入时机全靠直觉，没有系统化标准

### 核心功能

**3.1 实时控制面板**
人类随时可以看到整个团队的状态：
- 每个 Agent 当前在做什么、进度多少
- 待处理的决策请求（需要人拍板的事）
- 近期异常和 alert 汇总
- 团队整体健康度

**3.2 介入触发机制**
不靠人实时盯，靠系统主动推：

| 触发类型 | 条件 | 推送内容 |
|---|---|---|
| 不确定性触发 | 决策不可逆 / 多方案无法判断 | 选项 + 分析，人拍板 |
| 异常触发 | 验收未通过 / 输出超出范围 | alert + 建议处理方式 |
| 节奏触发 | 定时进展摘要 | exception only，不是流水账 |
| 主动升级 | Agent 发现自己能力边界 | 说明卡点，请求指导 |

**3.3 实时中断机制（解决消息队列痛点）**
当前最大的痛点：叫停指令因消息队列延迟而失效。

解决方向：
- **优先级消息通道**：人的中断指令走独立高优先级通道，不排队
- **执行检查点**：Agent 在每个任务节点主动 check 是否有中断指令，而不是跑完再看消息
- **任务状态锁**：收到中断后立即锁定状态，防止重复执行和消息风暴
- **幂等保障**：同一指令多次收到只执行一次

**3.4 Agent 互相验收**
质量保障不靠人肉 review：
- 每个任务完成后，质检 Agent 独立验收
- PM Agent 验收 Dev 的输出是否符合需求
- 异常自动上报 + 建议修复方案
- 人只需要处理 Agent 无法自己解决的问题

**3.5 协作透明度**
人可以随时追溯 Agent 之间发生了什么：
- 完整的 Agent 间通信记录
- 每个决策的来龙去脉
- 任务变更历史（谁改了什么、为什么改）

**3.6 授权分级**
明确定义什么可以 Agent 自主，什么需要人：
```
完全自主 → 日常执行、格式化输出、信息收集
需要确认 → 对外发布、资源消耗、影响他人的决策
必须人决 → 战略方向、预算、核心承诺
```

### 成功标准
> 人类每天只需要 1-2 次有效介入；出现问题时，系统在人类发现之前已启动修复

## Roadmap 建议

### Phase 1 — 解决自己的问题（MVP）
**目标：** 把我们自己的碳硅协作系统产品化，让另外 10 个人能用

优先级最高的功能：
- P0：**规则持久化与可信机制**（Agent 学到的东西可信、可见、可追溯）
- P0：反馈闭环系统（每次 feedback 都沉淀）
- P0：实时中断机制（解决消息队列痛点）
- P0：结构化 Onboarding 模板
- P1：角色模板库（5 个基础角色）
- P1：记忆体系标准化
- P2：控制面板 MVP

### Phase 2 — 扩展到早期用户
**目标：** 找到 50 个 information worker 验证产品价值

- 工具接入市场
- 团队组合向导
- 能力成长追踪
- 介入触发机制完整版

### Phase 3 — 规模化
**目标：** 定义 Hybrid Workforce 这个品类

- 社区贡献的 skill 和角色模板
- 企业版（团队共享 Agent 员工）
- Harness Engineer 工具链

## 我们的独特优势

1. **自己就是用户** — 每个功能都有真实场景验证，不是凭空设计
2. **先驱者视角** — 这个领域还很空白，我们在定义品类，不是跟随竞品
3. **完整的实践案例** — 周末搭子项目从 0 到 MVP 的全过程，是最好的 demo
4. **踩过的坑都是壁垒** — 消息队列、记忆丢失、角色边界……每个痛点的解法都是护城河

## 技术约束（交开发参考）

- **优先免费/开源方案**，不引入付费 SaaS 依赖
- 底层 LLM 按量付费可接受，基础设施层必须开源
- 具体选型由开发负责，PM 不介入
