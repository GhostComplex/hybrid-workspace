# Hybrid Workforce — 产品概念文档 v0.1

> **核心命题：** The rise of the Hybrid Workforce — where carbon and silicon collaborate as equals
>
> **一句话：** Build your workforce, not your tools.

---

## 核心叙事

> 过去 100 年，组织的基本单位是"人"。
> 未来 10 年，组织的基本单位是"碳硅团队"。
>
> 这不是 AI 工具的升级，这是**组织形态的进化**。

**我们的产品是这场进化的基础设施。**

### 为什么是现在？

四层技术同时成熟，第一次让"AI 作为真正的团队成员"变成可能——而不只是工具。

工业时代 → 信息时代 → **Agent 时代**

每一次转型都重新定义了"工作"的本质。Agent 时代的核心不是自动化，而是**涌现**——人机协作产生的能力远超双方之和。

未来的竞争力不是"会不会用 AI"，而是**"能不能管理一支 AI 团队"**。

新职业物种正在出现：
- **Harness Engineer** — 设计 Agent 行为边界和协作规则
- **AI Team Lead** — 管理碳硅混编团队，懂得何时授权、何时介入
- **Context Architect** — 构建组织记忆，让知识不依赖任何单一个体

## 核心设计原则：流程不是配置出来的，是学会的

**传统 workflow 工具：** 人配置流程 → 工具执行
**Hybrid Workforce：** 人描述意图 → Agent 自己理解、内化到行为准则 → 下次自动执行

> **"你不配置流程，你只需要告诉团队一次。"**

真实场景：Juanjuan 用自然语言描述一个 process，每个 Agent 各自领会自己在这个 process 中需要做什么，写进自己的行为准则或 skill 里，下次执行就自己学会了。

这意味着：
- **没有 BPMN 图，没有拖拽节点，没有 if-else 配置**
- Agent 的行为准则本身就是"活的流程文档"
- 可视化界面是展示 Agent 当前学会了哪些规则，而不是配置入口
- 产品的核心 UX 是一个**对话框**，用自然语言告诉团队怎么工作

## 核心挑战：Agent 怎么保证不忘？

### 问题本质
用户凭什么相信 Agent 真的记住了？

"我已写入行为准则"——这句话说出来容易，但如果只存在对话 context 里，下次对话窗口一关就消失了。这是碳硅协作里最基础的信任问题。

### 我们的实践给出了答案

**第一层：写入文件，不存在脑子里**
规则不能只停留在对话里，必须真正写进行为准则文件。文件永久存在，每次 Agent 启动都重新读取——就像人记笔记，记在脑子里会忘，写下来就不会。

**第二层：Memory 体系兜底**
行为准则记"应该怎么做"，daily memory 记"之前发生了什么"。两层叠加，Agent 既有规则，也有上下文，新对话开始时自动加载，跨天不失忆。

### 真实的坑
这两件事现在都是 Agent 自己手动去做的，没有强制保障：
- Agent 可能说"已写入"但实际没写
- 写进去了但格式混乱，下次读取时被忽略
- 多个 Agent 同时更新规则时产生冲突

### 产品需要解决的
**规则持久化的可验证机制：**
1. Agent 更新行为准则后，系统自动确认文件已写入
2. 规则在 Agent Profile 里可见——用户能看到这条规则是什么时候学的
3. 规则被实际触发执行时，有记录可追溯（"这条规则上次生效是 3 天前"）
4. 多 Agent 规则冲突时，有仲裁机制

> 这是 **Workforce Trainer 模块**最核心的功能——不是让 Agent 学得更快，而是让学到的东西真正**可信、可见、可追溯**。

## 场景一：建立 Agent 团队

### 问题本质
搭一套可用的 Agent 团队门槛极高：要懂 prompt engineering、要选工具、要设计角色分工。普通 information worker 根本搞不定，全靠自己摸索。

### 用户痛点
- 不知道需要哪些角色，从零开始设计太难
- 工具太碎片化，LangGraph / CrewAI / OpenClaw 各有一套逻辑
- 角色配置好了，Agent 之间怎么协作、谁给谁派活完全没有标准

### 解决方向

**1. 预设角色模板库**
像 Notion 模板一样，提供开箱即用的 Agent 角色包：

| 角色 | 职责 | 典型场景 |
|---|---|---|
| 研究员 Agent | 信息收集、市场调研、竞品分析 | 新项目启动前的调研 |
| PM Agent | 需求分析、PRD、优先级决策 | 产品规划 |
| 执行 Agent | 编码、内容生成、数据处理 | 日常执行任务 |
| 质检 Agent | 验收、审查、异常发现 | 每个任务的最后一关 |
| EM Agent | 进度追踪、blocker 清除、团队协调 | 多 agent 并行时的协调层 |

**2. 协作规则标准化**
定义清楚 Agent 团队的基本运作规则：
- 谁给谁派活（避免越级）
- 决策升级路径（Agent 不确定时找谁）
- 任务格式统一（每个任务建文件夹、过程留文档）

**3. 工具层不绑定**
角色定义与底层实现解耦，支持按需扩展工具能力，不锁定任何平台。

### 成功标准
> 一个没有 AI 工程背景的 PM，30 分钟内搭起一支可以开始工作的 Agent 团队

## 场景二：培训 Agent

### 问题本质
Agent 不是一次配好就完事。它需要持续 feedback、纠偏、能力扩展——这个过程完全没有系统化方法，全靠人肉摸索，耗时最多，也是人最容易放弃的地方。

这本质上是人力资源里最贵的三件事：**入职培训、绩效反馈、能力发展**。

### 用户痛点
- Agent 犯错了不知道怎么纠正，改了 prompt 又破坏其他地方
- 好的经验无法沉淀，下次遇到同类问题又从头来
- Agent 越用越让人不放心，而不是越用越好

### 解决方向

**1. 结构化 Onboarding**
每个 Agent 上岗前完成标准化入职：

```
角色定义（我是谁）
  → 行为准则（我怎么做事）
  → 边界清单（什么不是我的活）
  → 工作方式（任务格式、决策规则、汇报路径）
  → 记忆初始化（我需要知道哪些背景）
```

**2. 反馈闭环机制**
Agent 培训的核心不是写更好的 prompt，而是建立**反馈闭环**：

```
踩坑 / 收到 feedback
    → Agent 反思（reflection）
    → 更新行为准则
    → 内化为默认行为
```

工具：Reflection Agent pattern + 自动规则更新

**3. 能力扩展机制**
Agent 的能力边界可以按需扩展，且扩展后变成持久能力：
- 按需安装新工具（browser-use / 爬虫 skill / PPT 生成等）
- 主动学习：Agent 自己搜索、整理、沉淀领域方法论
- 知识库积累：RAG + 向量数据库，经验越用越多

**4. 记忆体系**
Agent 的"组织记忆"要比任何一个个体都厚：

| 记忆类型 | 内容 | 载体 |
|---|---|---|
| 短期记忆 | 当前任务 context | Session / 对话窗口 |
| 日常记忆 | 每天工作记录 | daily memory 文件 |
| 项目记忆 | 决策、进展、资料 | tasks/ 文件夹 + Git |
| 组织记忆 | 角色、联系人、规则 | contacts / projects / 方法论 |

### 成功标准
> Agent 每次犯错后，同类错误再发生率下降 80%；新 Agent 加入团队后，30 分钟内可以独立开始工作

## 场景三：优化碳硅协作流程

### 问题本质
什么时候人要介入？什么时候放手让 Agent 跑？现在完全靠直觉，没有任何框架。人介入太多 = 没有发挥 Agent 价值；介入太少 = 出错了没人发现。

### 用户痛点
- 不知道 Agent 在干什么，不放心但又不想一直盯着
- Agent 出问题时自己发现不了，等人类发现已经晚了
- 人和 Agent 之间职责边界模糊，什么都要问人，什么都要人拍板

### 解决方向

**1. 介入触发机制**
不靠人实时盯，靠系统触发：

| 触发类型 | 条件 | 人的动作 |
|---|---|---|
| 不确定性触发 | 决策不可逆 / 多方案无法判断 | 给选项，人拍板 |
| 异常触发 | 验收未通过 / 输出超出范围 | 收到 alert，决定是否介入 |
| 节奏触发 | 定时进展摘要 | 只看 exception，不看过程 |

**2. Agent 互相验收**
质量保障不靠人肉 review，靠 Agent 之间互相验证：
- PM Agent 验收 Dev 的输出
- 质检 Agent 独立跑验收 checklist
- 异常自动上报，而不是等人发现

**3. 授权分级**
明确定义哪些决策 Agent 可以自主，哪些需要人介入：

```
完全自主：日常执行任务、格式化输出、信息收集
需要确认：对外发布、资源消耗、影响其他人的决策
必须人决：战略方向、预算、核心用户承诺
```

**4. EM 协调层**
像 Scrum Master 一样，有一个 Agent 专门负责：
- 定时检查各 Agent 进展
- 发现 blocker 主动 @ 相关人
- 生成进展摘要推给人类负责人
- 人只需要在关键节点介入，不用全程在线

### 成功标准
> 人类成员每天只需要 1-2 次有效介入，Agent 团队其余时间自主运转；出现问题时，系统在人类发现之前已经启动修复流程

## 四层概念框架

| 层 | 概念 | 类比 |
|---|---|---|
| **Harness Layer** | Agent 的工作环境定义——行为准则、权限边界、工作流约束 | 公司的管理制度 |
| **Memory Layer** | 持久化的组织记忆——决策记录、经验沉淀、上下文传承 | 公司的知识库和文化 |
| **Learning Layer** | Agent 从纠正中自我进化——犯错→反思→规则更新→不再重犯 | 员工成长体系 |
| **Orchestration Layer** | 碳硅之间的动态分工——谁做什么、何时交接、卡住了找谁 | 项目管理 |

这四层对应产品的三个模块：
- Harness + Memory → **Workforce Builder**（搭团队）
- Learning → **Workforce Trainer**（培训 Agent）
- Orchestration → **Workforce OS**（协作运作）
